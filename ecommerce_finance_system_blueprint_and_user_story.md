# 电商财务系统蓝图与用户故事

## 一、系统目标

### 1. 统一底账
所有平台、地区、产品、订单类型的数据，最终都要落到同一套口径中。

系统以**平台结算报告**作为财务事实底账，确保各类分析、核对和月结都基于统一来源，而不是多个 Excel 或多套口径并行。

### 2. 自动出财务结果
原始数据上传后，系统应自动生成以下关键结果：

- 利润表
- 平台应收款核对结果
- 库存核对结果
- 多维经营分析报表
- 管理看板指标

目标是将财务从“人工拼表”转为“系统自动生成 + 人工审核例外”。

### 3. 过程可稽核
每一个财务结果都必须可以追溯到：

- 原始上传文件
- 原始字段
- 口径映射规则
- 计算规则版本
- 调整记录
- 人工确认记录
- 审批记录

系统不仅要算得出来，还要经得起复核和审计。

### 4. 月结可控制
系统必须具备严格的月结检查机制。

在关账前，系统自动执行多维度校验；若发现关键差异且未被合理解释或审批，则：

- 系统报错
- 阻止关账
- 保留异常台账与处理痕迹

目标不是“生成报表”，而是“确保财务结果准确后才能关账”。

## 二、核心设计原则

### 1. 结算报告是财务事实主账
订单报表反映业务发生，结算报表反映平台最终确认并准备结算的财务结果。

因此系统应明确：

- **Settlement Ledger（结算主账）** = 财务底账
- **Order Detail Layer（订单明细层）** = 业务分析与穿透
- **Inventory Layer（库存层）** = 成本与库存核对
- **Validation Layer（校验层）** = 验证与解释差异

其他报表不能替代结算主账，只能辅助、补充、验证。

### 2. 先统一业务语言，再做系统
不同平台对销售、退款、佣金、仓储费、广告费、结算金额、回款等字段定义并不一致。

系统必须建立一套统一的数据语义层 / 财务数据字典，将各平台原始字段映射为标准财务口径，例如：

- `gross_sales`：毛销售额
- `sales_discount`：销售折扣
- `refund_amount`：退款金额
- `net_sales`：净销售额
- `platform_commission`：平台佣金
- `fulfillment_fee`：履约费
- `storage_fee`：仓储费
- `advertising_fee`：广告费
- `settlement_amount`：结算金额
- `platform_receivable`：平台应收
- `cogs`：销售成本
- `gross_profit`：毛利
- `net_profit`：净毛利
- `roi`：投资回报率

没有统一语义层，就不会有统一利润表和统一核对逻辑。

### 3. 既能自动化，也能容纳人工判断
财务系统中，永远存在三类事项：

- **全自动可计算**：如订单量、销售额、平台佣金汇总
- **有规则可判断**：如按规则自动归类的费用或异常
- **必须人工判断**：如争议退款、赔偿归因、跨期差异、特殊费用归属、库存差异责任认定

因此系统必须设计 **Manual Review Workbench（人工审核工作台）**。

对无法自动判断的事项，系统应：

- 自动生成待处理任务
- 提供系统建议处理方式
- 允许财务在前台完成确认
- 保留备注、附件、审批与审计痕迹

### 4. 关账控制比报表更重要
成熟的财务系统，核心不在于“报表很多”，而在于：

**当底层逻辑没有对平时，系统必须拦住用户，不允许关账。**

因此系统必须具备：

- 关账前检查清单
- 自动比对规则
- 差异阈值管理
- 例外审批机制
- 报错分级机制
- 关账状态机

财务系统真正的价值，不只是生成结果，而是防止带错结果进入财务结账。

## 三、系统整体架构

系统建议由以下 10 个模块构成。

### 模块 1：数据接入中心
负责接收各平台原始文件和辅助文件。

**支持的数据源包括：**

1. 平台结算报告  
   - Amazon Settlement / Transaction  
   - TikTok Shop Settlement  
   - Shopify / 独立站 Payout / Payment / Order  
   - 抖店结算单  
   - 京东结算报表  
   - 后续新增平台

2. 订单明细报告  
   - 用于颗粒度展示  
   - 用于与结算数据交叉验证

3. 库存与成本数据  
   - ERP / WMS 出入库  
   - 采购入库  
   - 调拨  
   - 盘点  
   - 期初期末库存  
   - SKU 成本数据

4. 费用数据  
   - 广告费用  
   - 仓储费用  
   - 物流费用  
   - 平台外费用  
   - 手工调整数据

5. 主数据  
   - SKU 主数据  
   - 产品名映射  
   - 平台映射  
   - 地区映射  
   - 店铺映射  
   - 订单类型映射  
   - 科目映射  
   - 币种与汇率

### 模块 2：主数据与映射中心
这是系统成败关键，用于统一业务语言。

**主数据对象包括：**

- 地区
- 国家
- 平台
- 店铺
- SKU
- SPU
- 产品线
- 订单类型
- 费用类型
- 币种
- 结算周期
- 平台交易类型
- 财务科目

**映射能力包括：**

- 平台字段 → 统一字段
- 平台交易类型 → 财务交易类型
- SKU 别名 → 标准 SKU
- 产品名多语言 / 多平台别名 → 标准产品
- 平台店铺 → 法人主体
- 平台费用项 → 财务费用科目
- 订单类型 → 财务分析标签

### 模块 3：结算主账引擎
这是系统核心，用于把平台结算数据沉淀成标准化财务事实记录。

**建议标准字段包括：**

- 结算单号
- 平台
- 店铺
- 地区
- 国家
- 法人
- 币种
- 结算期间
- 订单号
- 子订单号
- SKU
- 产品
- 订单类型
- 交易类型
- 销量
- 订单数
- 毛销售额
- 折扣
- 退款
- 净销售额
- 平台佣金
- 履约费
- 仓储费
- 广告费
- 其他费用
- 结算金额
- 应收金额
- 成本
- 毛利
- 净毛利
- 数据来源文件
- 上传批次号
- 规则版本
- 审核状态
- 调整标记

**该模块需要支持：**

- 多平台格式兼容
- 重跑机制
- 原始文件追溯
- 版本管理
- 手工调整留痕
- 按月冻结

### 模块 4：校验与稽核引擎
系统不仅要计算，还要验证。

**校验分为 4 层：**

#### A. 完整性校验
- 是否缺结算文件
- 是否缺订单明细
- 是否缺库存成本
- 是否缺汇率
- 是否缺 SKU 映射
- 是否重复上传
- 是否存在空关键字段

#### B. 一致性校验
- 结算销量 vs 订单销量
- 结算销售额 vs 订单销售额
- 结算退款 vs 退款明细
- 平台应收 vs 回款记录
- 发货数量 vs 成本出库数量
- 订单 SKU vs 主数据映射结果

#### C. 财务逻辑校验
- 净销售额不能异常为负
- 平台佣金率异常报警
- ROI 异常波动报警
- 单 SKU 毛利率超阈值报警
- 结算金额与构成项加总不符时报错
- 跨期费用归属异常
- 缺成本却生成利润时报错

#### D. 关账控制校验
- 数据上传完成
- 关键映射完成
- 关键异常已处理
- 平台应收核对完成
- 库存核对完成
- 利润表锁定完成
- 例外项已审批

### 模块 5：人工确认与例外处理中心
用于承接系统无法自动判断的事项。

**典型场景包括：**

- 无法自动识别的 SKU 映射
- 特殊费用归类
- 平台赔偿 / 罚款归属
- 广告费分摊方式
- 订单类型归类异常
- 跨月差异归属
- 库存盘亏盘盈原因判断
- 平台应收差异解释

**前台页面应包含：**

- 异常类型
- 来源平台 / 地区 / 店铺
- 涉及订单 / SKU / 金额
- 系统建议处理方式
- 财务人工选择项
- 备注说明
- 附件上传
- 审批流
- 操作留痕

### 模块 6：利润表引擎
用于自动生成电商经营利润分析结果。

**支持的分析维度包括：**

- 地区
- 国家
- 平台

- 产品名
- SKU
- 产品线
- 订单类型

- 销售成本

- 履约费
- 仓储费
- 广告费
- 其他费用
- 毛利
- 净毛利
- 毛利率
- 净毛利率
- ROI
- 客单价
- 单件成本
- 单件净毛利

### 模块 7：平台应收核对中心
用于实现平台应收与回款自动核对。

**核对逻辑：**

> 期初应收 + 本期应收发生 - 本期回款 = 期末应收

**关键输出包括：**

- 平台应收账龄
- 已结算未回款
- 已回款未匹配
- 差异金额
- 差异原因分类
- 核销状态

### 模块 8：库存核对中心（非核心业务，待实现）
用于连接库存、成本与利润。

**系统应支持：**

- 期初库存
- 本期入库
- 本期出库
- 调拨
- 退货回库
- 盘盈盘亏
- 期末库存

**关键核对关系：**

- 销售出库数量 vs 销量
- 成本结转数量 vs 发货数量
- 期末库存账面 vs 仓库系统
- SKU 成本口径是否一致
- 库存调整是否审批

### 模块 9：报表中心
用于统一输出可下载的财务与经营报表。

**支持的筛选维度：**

- 地区
- 平台
- 店铺
- 产品名
- SKU
- 订单类型
- 时间范围
- 币种
- 法人主体

**输出形式：**

- 表格报表
- 明细导出
- 汇总导出
- 固定模板下载
- 自定义列下载

### 模块 10：管理看板与月结控制中心
用于展示关键指标和管理关账状态。

**建议看板层级：**

1. CFO / 管理层看板  
   - 销量  
   - 订单数  
   - 净销售额  
   - 净毛利  
   - ROI  
   - 平台回款  
   - 库存余额  
   - 关账状态  
   - 重大异常数量

2. 财务运营看板  
   - 待核对平台数  
   - 应收差异金额  
   - 库存差异金额  
   - 未处理异常数  
   - 待人工确认事项  
   - 关账进度

3. 业务分析看板  
   - 平台表现  
   - 产品表现  
   - 地区表现  
   - 订单类型表现  
   - 毛利率变化  
   - ROI 变化

**建议月结状态包括：**

- 未开始
- 数据待上传
- 数据处理中
- 映射待确认
- 异常待处理
- 应收待核对（先已手工填入实现，后续可以优化OCR识别，PDF转CSV)
- 库存待核对
- 利润表待确认
- 审批中
- 已关账
- 已重开（重开之后回到“异常待处理”状态）

## 四、用户故事

### 4.1 角色背景
我是电商公司的财务人员，负责多个平台、多个地区的销售、结算、库存、成本、应收和月结核对工作。

目前公司经营的平台包括但不限于：

- 美国市场：Amazon、TikTok Shop、独立站
- 中国市场：抖店、京东
- 未来可能新增其他国家、地区及平台

我希望建设一套统一的财务系统，以平台结算报告作为底层数据库，其他报告均围绕结算主账进行补充、验证和解释，从而自动生成利润表、库存核对结果和平台应收款核对结果，并形成清晰、可审计的稽核与月结环境。

### 4.2 核心用户故事

作为一名电商公司财务，  
我希望把多个平台、多个地区的销售数据、库存数据、费用数据和回款数据统一放入一个系统中，  
并且以平台结算报告作为唯一财务底账，  
这样我只需要上传原始数据，系统就能够自动生成利润表、库存核对结果、平台应收款核对结果以及多维经营分析报表，  
同时还能展示完整的检查、校验、差异处理和关账过程，  
从而保证所有财务结果准确、统一、可追溯，并支持月结控制。

### 4.3 场景化用户故事

#### 用户故事 1：统一上传并生成底账
作为财务人员，  
我希望把 Amazon、TikTok Shop、独立站、抖店、京东等平台下载的原始结算报告直接上传到系统，  
这样系统就可以自动解析、映射并生成统一的结算主账，  
从而避免我在 Excel 中手工整理不同平台格式的数据。

#### 用户故事 2：统一辅助数据来源
作为财务人员，  
我希望订单报表、库存报表、成本报表、广告报表、回款报表等辅助数据也能进入同一系统，  
并且这些数据源都与结算主账保持统一口径，  
这样我就可以确保各类分析和核对都来自同一套数据基础，而不是各自独立计算。

#### 用户故事 3：多维查看销售与利润
作为财务人员或管理者，  
我希望能够按地区、平台、产品名、SKU、订单类型、时间等维度，自主筛选查看销售情况，  
包括销量、订单数、净销售额、成本、各类费用、净毛利、ROI 等指标，  
这样我就可以快速查看各平台、各产品、各地区的经营表现，并导出报表。

#### 用户故事 4：看板查看关键指标
作为管理层或财务负责人，  
我希望通过看板的形式按地区、平台、产品或订单类型查看关键指标，  
包括销量、订单数、净销售额、净毛利、ROI、平台回款、库存余额等，  
这样我就可以快速掌握经营表现和财务风险，而不需要逐张查看明细报表。

#### 用户故事 5：人工确认例外事项
作为财务人员，  
我希望对系统无法自动判断的事项，在系统前台进行人工确认，  
例如 SKU 映射异常、特殊费用归类、赔偿归因、库存差异原因、应收差异说明等，  
这样我就可以在系统内完成必要判断，并保留完整操作记录和审批痕迹。

#### 用户故事 6：自动核对平台应收
作为财务人员，  
我希望系统根据结算数据和回款数据自动核对平台应收款，  
并展示期初应收、本期新增、本期回款、期末应收及差异原因，  
这样我就可以快速发现未回款、未匹配和异常差异，提升平台账务核对效率。

#### 用户故事 7：自动核对库存与成本
作为财务人员，  
我希望系统将库存、出入库、销售出库和成本结转数据统一联动，  
自动生成库存收发存、库存差异和成本异常结果，  
这样我就可以判断利润结果是否建立在正确的库存和成本基础上。

#### 用户故事 8：月结前自动检查
作为财务经理，  
我希望系统在月结前自动进行多维度检查，  
包括数据完整性、一致性、财务逻辑、平台应收核对、库存核对和异常处理状态，  
这样我就可以在关账前发现问题并及时处理，而不是关账后再返工。

#### 用户故事 9：阻断式关账
作为财务负责人，  
我希望当关键校验未通过时，系统必须报错并阻止关账，  
只有在差异处理完成或经过合理审批后才允许关闭账期，  
这样我就可以确保最终利润表和核对结果的准确性，避免带错关账。

#### 用户故事 10：全过程可追溯
作为财务人员、管理者或审计复核人员，  
我希望系统中每一个利润、应收、库存和差异结果都可以追溯到原始文件、字段、规则和人工处理记录，  
这样我就可以快速完成复核、解释、审计和管理追责。

## 五、总结

这套系统的本质，不是一个普通报表系统，而是一个：

**以结算主账为核心、以校验引擎为护栏、以利润与应收闭环为第一目标、以库存闭环为第二目标的电商财务控制平台。**

一句话总结：

> 先做准，再做全，再做快。

系统首先要解决的是财务准确性和关账可控性，再逐步扩展经营分析、库存闭环和全球化能力。
