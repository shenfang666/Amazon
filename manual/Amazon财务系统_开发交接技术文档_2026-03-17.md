## 2. 系统定位

系统名称：Amazon 财务闭环中心

核心定位：围绕 Amazon 平台财务月结，提供“利润、应收、异常、库存、月结状态、数据接入、数据导出”的统一操作平台。

当前系统已经不只是报表展示，而是一个轻量工作流系统，覆盖：

- 数据接入
- ETL 装载与衍生视图
- 异常识别与人工判断
- 月结状态推进
- 最细颗粒度数据预览与下载
- 手工控制表落库与留痕

## 3. 当前技术栈

后端：

- Python 3.14
- 标准库 HTTP 服务：`http.server.ThreadingHTTPServer`
- SQLite：单库部署

前端：

- 原生 HTML / CSS / JavaScript
- 无 React/Vue，无构建步骤
- 静态页面 + 运行时动态增强

数据处理：

- 一组 Python ETL 脚本
- 通过 SQLite 表 + SQL 视图完成大部分财务计算

部署方式：

- 本地单进程运行 `app.py`
- 数据库存放在项目根目录 `amazon_finance.db`

## 5. 实际运行形态

### 5.1 运行入口

启动命令：

```powershell
python app.py --host 127.0.0.1 --port 8000
```

或：

```powershell
python app.py --host 127.0.0.1 --port 8766
```

服务启动后：

- 首页由 `DashboardHandler` 提供
- 静态资源来自 `web/`
- 动态增强脚本来自 `/runtime-app.js`

### 5.2 静态页面与运行时页面不完全一致

这一点非常重要。

当前系统不是“只靠 `web/index.html + web/app.js`”渲染。真实运行时还会通过：

- `app.py` 中的 `render_index_html()`
- `app.py` 中的 `RUNTIME_APP_JS`

动态插入或调整：

- 库存核对页签
- 订单追踪独立页签
- 数据库页签
- 上传治理面板
- 异常工作台增强交互
- 最细颗粒度数据库明细入口
- 异常分组汇总、明细筛选、按类别提交、移除费处理表单

因此：

- 不要只根据 `web/index.html` 判断最终页面结构
- 不要只根据 `web/app.js` 判断最终交互行为
- 运行态行为必须结合 `app.py` 一起看

## 12. 关键业务域说明

### 12.1 利润域

利润主口径基于月度汇总视图：

- `v_monthly_sku_order_type_summary`

支持：

- SKU 粒度利润分析
- 订单类型汇总
- 最细颗粒度行级明细
- 利润口径 / 应收口径双口径投影

最细明细的核心函数：

- `build_order_line_profit_rows`
- `project_order_line_rows`

### 12.2 订单追踪域

核心视图：

- `v_order_settlement_tracking`

支持：

- 订单与 SKU 查询
- 结算状态过滤
- 订单财务明细定位

当前已从“利润分析”中独立为单独页签。

### 12.3 应收域

应收快照由以下逻辑生成：

- 开始应收：取上月 closing_receivable
- 本期应收：汇总利润视图中的应收口径字段
- 本期回款：取 `fact_platform_receipts`
- 差异：`opening + current - receipts`

当前最重要的现实问题：

- `fact_platform_receipts` 为空
- 因此 `receivable_reconciliation_pending` 会持续出现

### 12.4 异常域

异常分为两类：

- 系统生成异常
- 人工工单

系统生成异常的实际展示逻辑已升级为：

- 按异常类型分组
- 可查看异常条数
- 可查看异常明细
- 可按类别统一提交“正常差异/待处理异常”
- 明细支持账期筛选与合计金额

特殊规则：

- 异常列表不再只看“最近一次检查窗口”
- 当前口径是：未关账期间所有仍存在的异常

### 12.5 移除费控制域

`pending_removal_control` 是当前异常工作台最重要的业务型异常之一。

后端表：

- `manual_removal_fee_controls`

前端现状：

- 已支持在异常明细中直接录入
  - `removal_category`
  - `accounting_treatment`
  - `source_note`

录入后行为：

- 写入 CSV：`manual/manual_removal_fee_controls.csv`
- 写入表：`manual_removal_fee_controls`
- 重建视图
- 重跑受影响月份的月结检查
- 当前页明细立即移除对应记录

### 12.6 库存域

库存模块已具备：

- 库存流水录入
- 快照生成
- 库存状态判断
- 手工调整留痕

但当前库内：

- `fact_inventory_movements = 0`
- `fact_inventory_snapshot = 0`

因此：

- 库存模块逻辑完备
- 但目前仍属于“功能已就绪，数据未接入”

### 12.7 月结状态机

状态包括：

- `waiting_upload`
- `processing`
- `mapping_pending`
- `exception_pending`
- `receivable_pending`
- `inventory_pending`
- `pnl_pending`
- `approving`
- `closed`
- `reopened`

推进逻辑：

- 若有 blocker：不能进入关账
- 若应收未平：停在 `receivable_pending`
- 若库存未就绪：停在 `inventory_pending`
- 条件满足后：可进入 `approving`，再 `closed`

注意：

- 月结 blocker 采用“有效异常”口径
- 已被人工认定为正常差异的异常不会继续阻断

## 14. 当前性能状态

2026-03-17 实测：

- `build_dashboard_payload('2026-02')`：约 3.5 秒
- `build_profit_payload('2026-02')`：约 1.5 秒
- `build_exceptions_payload('2026-02')`：约 0.03 秒
- `build_month_close_payload('2026-02')`：约 0.02 秒

本轮已完成的性能优化：

- `get_months()` 从重视图改为基础表联合取数
- 异常页移除了大量重复重查询
- 库存状态改成按需刷新
- 利润分析/订单追踪改成按页签懒加载

剩余瓶颈：

- 首页总览仍有约 3 秒级耗时
- 主要慢在：
  - `build_overview`
  - `trend` 汇总
  - `top_skus` 汇总

若继续优化，优先建议：

1. 为总览做月度缓存表或物化快照
2. 减少首页即时聚合 SQL
3. 视情况给 SQLite 增加索引或拆分为汇总表

## 15. 当前已知问题与风险

### 15.1 `fact_platform_receipts` 为空

风险：

- 应收 warning 长期存在
- `receivable_reconciliation_pending` 无法自然消失

建议：

- 优先补通 15 号脚本对应的收款数据源

### 15.2 库存模块无真实数据

风险：

- 库存页可用但缺乏实际业务闭环验证

建议：

- 补充库存数据源或导入模板

### 15.3 `RUNTIME_APP_JS` 过大

风险：

- 页面真实行为分散在 `web/app.js` 与 `app.py`
- 接手人不容易定位

建议：

- 分阶段把运行时增强回收进静态前端

### 15.4 `app.py` 单文件过大

现状：

- 约 19 万字符
- 业务域全部耦合在同一文件

风险：

- 改动冲突大
- 新人理解成本高
- 回归测试困难

建议拆分：

- `services/exceptions.py`
- `services/month_close.py`
- `services/inventory.py`
- `services/receivables.py`
- `services/exports.py`
- `server/routes.py`

### 15.5 历史异常与人工工单较多

现状：

- `manual_exception_case` 历史记录已很多
- 当前页面已把它下沉到底部，作为历史区

风险：

- 如果继续增长，需考虑分页或归档
